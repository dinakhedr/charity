Core Design Principles
Soft-Delete Architecture
Every record has IsDeleted, DeletedAt, DeletedBy
Default views hide deleted records
"Show Deleted" toggle (if canViewDeleted)
Restore action logs to AuditLog sheet
Full audit trail: CREATE, EDIT, DELETE, RESTORE tracked with timestamp & user
RBAC Permissions Matrix
Permissions sheet stores: CanView, CanCreate, CanEdit, CanDelete, CanViewDeleted, CanViewSensitive, CanExport
Roles: SuperAdmin, Director, AccountManager, CaseManager, DataEntry, Viewer, InventoryManager
Sensitive fields hidden unless canViewSensitive = true
Permissions.html allows live overrides (SuperAdmin only)
Google Sheets Data Layer
Single spreadsheet with 20+ sheets
Operational: Transactions, Donors, Beneficiaries, Receipts, Inventory, Projects, Installments, Recurring
Lookup sheets: Categories, DonorTypes, IslamicClasses, NeedTypes, PaymentMethods, AidFrequencies, etc.
System: Permissions, AuditLog, Settings

Role hierarchy (example)

  • SuperAdmin — full access, user mgmt, view deleted
  • Director — full operational + soft-delete transactions
  • AccountManager — finance modules (Transactions, Receipts, Recurring)
  • CaseManager — Beneficiaries, Projects, view-only finance
  • DataEntry — create/edit on Donors/Beneficiaries/Transactions, no delete
  • Viewer — Dashboard + Reports only, read-only
